Why this configuration

The exponential wall: hardware comparators = trees × (2^depth − 1). Depth is exponential — depth 10 is ~33× the area of depth 5. That's why deep trees are a hardware trap. (They also overfit — we measured it — so we never wanted them anyway.)
Software doesn't care: a CPU/ESP32 walks the trees one node at a time from memory — cost is linear (trees × depth), not exponential. A huge model just takes a few more microseconds. So "fits in hardware" is a different question from "runs on a chip."
The trees are cheap; the features are expensive. Adding features barely changes the tree logic — but each feature must be computed in hardware (rolling means = shift registers; the messy ones need variable divisors + missingness tracking). And every feature you compute, you must verify bit-exact against Python. That verification cost — not silicon area — is the real reason we ship 4 clean features instead of 40.

Hardware numbers are exact. ROC-AUC is illustrative, anchored to two measured points: 4 features → 0.674, 40 features → 0.722 (5-fold patient CV). Both models fit real FPGAs; the edge model fits the cheapest one and is tractable to verify.
